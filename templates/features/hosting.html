传统多号托管的致命缺陷

  • ✕
    IP 关联导致封号潮

    多账号在同个 IP 下并发登录，一旦某个账号因违规被封，该 IP 下的其他账号会被 Telegram 自动关联进行封禁。

  • ✕
    多实例性能瓶颈与连接断开

    单节点并发运行上百个 Telethon 客户端时，CPU 和内存极易被占满，导致网络延迟升高、频繁掉线和登录过期。

  • ✕
    缺乏隔离的单机架构

    缺乏分片机制，一台机器异常宕机导致所有托管账号的业务全部瘫痪，造成不可挽回的私域流量流失。

我们的安全托管技术方案

  • ✓
    独享独立代理（物理隔离）

    支持为每个托管账号绑定专有的 Socks5/MTProto 网络代理。账号网络相互独立，规避任何关联封号风险。

  • ✓
    底层分布式号池分片（Sharding）

    通过 POOL_TOTAL_SHARDS 实现多实例分片运行。每个分片独立拉取任务，支持平滑横向扩展，轻松托管数千个账号。

  • ✓
    安全防漂移卫士机制

    系统内置 POOL_SHARD_GUARD。一旦检测到分片分配异常或配置冲突，自动安全挂起避免重复登录造成封号。

关于账号托管的常见疑问

1. 什么是 Socks5/MTProto 独立代理？

独立代理是将您的每个 Telegram 账号的网络流量单独分流，不与其他账号共用出口 IP。这样即使某个账号有敏感操作，其他账号也不会受波及，能有效降低批量封号概率。

2. 系统最多支持托管多少个账号？

由于底层采用了分布式分片架构（Sharding），支持横向扩展。您可以通过配置 POOL_TOTAL_SHARDS 轻松将账号负载分散在多台机器上，理论上支持无限数量的账号托管。

3. 软删除审计有什么作用？

为了防止误操作，我们的系统对账号采用“软删除”机制（Soft Delete）。当您在控制台移除某个账号时，系统并不会彻底清除数据，而是将其状态标记为已停用，方便后续进行审计和找回。

4. 托管时我的账号密码会被泄露吗？

系统仅将您的 session 会话安全存储在受隔离的本地或云存储中。我们绝不会传输您的密码或二次验证（2FA）密码，且账号的所有登录态和凭据均采用高强度加密管理。
