系统技术架构原理与运行机制

1. Telegram 多账号安全托管

系统基于高并发协程框架 Telethon 驱动，以异步非阻塞模式同时维护上百个账号的安全连接。为了防止 Telegram 官方封号，系统支持对每个账号配置独立的网络代理（如 Socks5、Socks4 或 HTTP 代理隧道），对每一个连接请求进行底层协议级别的 IP 隔离，模拟自然的用户终端网络特征，提升账号的高可用性与托管寿命。

2. 分布式号池集群自适应分片

在大规模的自动群发和关键词智能自动回复业务中，单机承载能力是有限的。系统通过数据库的 InstanceHeartbeat（实例心跳） 实现多机动态扩容与自适应负载分片机制。各节点启动后，将自身实例 ID 与心跳定时上报。在账号巡检轮次中，实例结合总分片数与当前分片索引，利用哈希取模算法动态切分账号，保证各个 worker 节点的流量均衡，并触发分片漂移防护。

3. APScheduler 高阶定时调度

定时发送与群发任务通过底层的 APScheduler（Advanced Python Scheduler） 调度器驱动。APScheduler 定期扫描数据库中的活动任务，以精确的 Cron 表达式解析下一次执行时间，并将任务封装为后台异步 Job。通过内存队列与线程池进行派发，保证高并发消息推送下的微秒级精确触发，实现极低的网络延迟和高投递成功率。

常见技术排障与系统运行问答 (FAQ)

Q1: 为什么有账号处于启用状态（Active）却未显示在线（Online）？

答：启用状态（Active）属于业务启用标记，表明该账号被允许参与托管和发送任务。在线状态（Online）则是号池轮询节点实际连接 Telegram 服务的会话状态。若账号刚刚添加，或遇到了 Telegram 凭据失效（Session Expired）、网络代理失效、以及账号被 Telegram 官方临时限制等问题，系统会在巡检失败后将在线状态置为 False，并通过日志归类相应的错误码，自动启动退避重试。

Q2: 调度器中的定时 Job 是如何实现自动恢复和容错的？

答：APScheduler 在启动时会通过 ReloadActiveTasksToScheduler 方法从 MySQL 数据库中一次性加载所有被启用的定时发送规则，并为它们重新计算下次发送时间。如果在运行期间发生宕机，当服务重新拉起后，调度器会自动计算发生崩溃时的“漏失时间”（Misfire），并根据容错档位参数判定是否需要补发或直接顺延到下一个触发时间，保证营销消息的不重不漏。

Q3: 号池分片漂移防御机制（Shard Drift Guard）的作用是什么？

答：在大规模部署时，如果节点网络发生闪断、数据库连接超时或容器漂移，可能会导致某台服务器的心跳停止数秒，从而被集群内其他存活实例判定为掉线而重新瓜分其分片账号。若此时原实例重新上线，就容易产生多实例同时托管和轮询同一账号的问题，导致冲突封号。Shard Drift Guard 开启后，一旦检测到分配给自己的账号与其他实例的分片范围产生交叉或漂移，会立即主动退出进程，从硬件级别杜绝双写和冲突。
